# Module Conversion Workflow

用于处理页面转换相关任务的工作流。

## 适用场景

当用户的描述中明确包含“转换类型”或“目标形态”时调用，例如：

- 把页面 A 转成页面 B
- 把 PC 页面转成移动端页面
- 把旧模块改成新模块结构
- 把某个页面转换为指定风格、模板或业务模块

## 工作目标

在不随意修改 UI 组件实现的前提下，优先通过外层编排、容器层、配置层、数据映射层和模块组合方式完成页面转换。

## 执行流程

1. 识别转换任务
   - 提取源页面、目标页面、转换类型、约束条件。
   - 确认是结构转换、样式适配、模块替换，还是数据流改造。

2. 明确边界
   - 优先判断是否允许修改现有 UI 组件。
   - 若用户要求“不改 UI 组件”，则仅在外层页面、容器、路由、配置、接口适配层实现转换。

3. 分析现状
   - 找到源模块入口、依赖组件、状态管理、接口调用和路由关系。
   - 区分可复用部分与必须新增的转换层。

4. 设计转换方案
   - 优先采用组合、包装、映射、适配方式，而不是直接重写底层组件。
   - 明确需要新增的页面容器、转换函数、配置项、接口适配器和路由调整。

5. 实施改造
   - 小步修改，优先保持现有行为稳定。
   - 仅在必要时补充最少量注释，解释转换层的职责。

6. 验证结果
   - 检查页面是否完成目标转换。
   - 检查是否引入类型错误、路由错误、接口映射错误或状态同步问题。

## 约束原则

- 禁止修改 `ui` 目录下的组件。
- 优先在外层控制页面转换逻辑。
- 保持现有交互和数据契约尽量稳定。
- 如果转换描述不完整，先补齐源模块、目标模块和转换类型再实施。

## 输出要求

执行该类任务时，输出应包含：

- 当前识别到的转换类型
- 建议采用的转换路径
- 是否需要新增适配层
- 是否涉及路由、数据结构或接口调整
- 如果开始编码，说明将修改哪些外层文件
