# 当前实现与基线 (Current Implementation & Baseline)

## 1. 系统架构 (System Architecture)
当前的 **DocFlow** 项目基于 **RAG (Retrieval-Augmented Generation)** 和 **云端推理** 的混合架构，具备轻量级部署、依赖少的特点。
- **前端 (Frontend)**：React 18 + TypeScript + Vite + Zustand + shadcn/ui。提供现代化的文件上传和对话交互界面。
- **后端 (Backend)**：Python 3.11 + FastAPI + Celery + Redis + asyncio。负责核心业务逻辑、并发控制和 RAG Pipeline。
- **AI/ML 层**：
  - **LLM 推理引擎**：默认走 OpenAI 兼容云端接口（降低本地硬件要求）。
  - **Embedding 模型**：支持 Gemini 和 OpenAI 兼容 Embedding 服务。
  - **RAG 框架**：LangChain（文本切片、加载、Prompt 构建）。
  - **向量数据库**：ChromaDB 0.5.0（本地部署，数据持久化存储）。

## 2. 核心功能现状 (Current Capabilities)
当前已经实现并可运行的核心功能如下：

1. **文档多格式解析与入库**：
   - 支持解析 PDF、Word (.docx)、Markdown 和普通文本文件。
   - 使用 `Unstructured` / `PyPDF` 等工具将长文档加载并按照固定 Chunk Size 切片。
   - 文档切片后通过 Embedding 模型生成向量，并持久化写入本地 ChromaDB 中。

2. **RAG 智能文档问答**：
   - 用户可以针对上传入库的文档发起提问。
   - 后端通过用户问题在 ChromaDB 中检索 Top-K 相关的文本切片（Context）。
   - 将检索到的 Context 连同用户问题，输入给云端 LLM，生成最终的答案。

3. **系统健康检查与探活**：
   - 提供专门的端点，实时检查 LLM、Embedding 模型和 ChromaDB 服务的连通性和可用性。
   - 方便用户在本地部署后快速排查网络、API Key 和配置问题。

4. **基础用户界面 (UI)**：
   - 提供了文件上传和解析进度展示的界面。
   - 提供了一个对话窗口，允许用户输入问题并获取 AI 的回答。

## 3. 当前系统的局限性与待办 (Current Limitations)
1. **基础功能不完善**：
   - 虽已实现对话，但缺乏完整的**文档管理能力**（文档列表查看、单个文档删除等操作未真实接通）。
   - **文档摘要**能力尚未完成，仍为占位状态（无法一键生成全文大纲或 Summary）。
2. **缺少高阶分析能力**：
   - **要点提取**（Action Items、风险点提取）等更结构化的总结分析尚未落地。
   - **多文档对比**功能尚未开发。
3. **并发与大规模处理能力弱**：
   - 目前主要设计为单实例部署（Semaphore 限制并发），对超大文件或多用户高并发的体验需要进一步优化。
4. **前端交互深度不够**：
   - 没有历史会话记录（Session 管理）。
   - 问答引用来源（Citations）尚未在 UI 上做高亮展示。
