# 核心功能需求 (Phase 1) - Core Features

基于当前的基线实现，**阶段一**的开发重点在于补齐基础功能的缺失，提供一个闭环的、体验优良的“文档阅读与管理系统”。

## 1. 文档管理与列表 (Document Management)
**目标：** 让用户可以直观地查看、管理已上传的知识库文档。
- **需求描述**：
  - **文档列表展示**：提供一个完整的知识库视图，展示已入库的文件名、大小、上传时间、解析状态（成功/失败/处理中）、文件类型图标。
  - **删除与清理**：支持用户删除单个文档。删除操作需同步清理底层 ChromaDB 中的对应向量切片以及本地缓存的原始文件。
  - **批量操作**：支持多文件同时上传、批量删除功能。
- **技术实现路径**：
  - 后端需要增加 `GET /documents`、`DELETE /documents/{id}` 接口。
  - ChromaDB 需要按 `metadata.source` 或 `metadata.document_id` 进行过滤删除。

## 2. 自动文档摘要生成 (Auto-Summarization)
**目标：** 用户无需阅读或提问，系统能在文档上传后自动生成一篇高价值的总结摘要。
- **需求描述**：
  - **全文大纲与核心观点**：自动生成结构化的大纲，提取作者想要表达的核心 3-5 点内容。
  - **执行摘要 (Executive Summary)**：提供 150 字左右的快速速览，方便用户 30 秒内了解文档主旨。
  - **UI 呈现**：在文档列表中，点击某个文档即可在侧边栏或弹窗中查阅摘要。
- **技术实现路径**：
  - 采用 **Map-Reduce** 或 **Refine** 的方式调用 LLM 对切片文本进行归纳。
  - 将生成的 Summary 作为 Metadata 或单独的 Collection 存入数据库。

## 3. 结构化要点提取 (Key Points Extraction)
**目标：** 针对特定类型的文档（如会议纪要、项目计划），提取结构化的高价值信息。
- **需求描述**：
  - **Action Items（待办事项）提取**：自动识别文档中指派给谁的任务、Deadline 和执行动作。
  - **风险点与决策**：提炼项目文档中的潜在风险及会议中的关键决策。
- **技术实现路径**：
  - 后端提供一个专门的接口 `/documents/{id}/extract`。
  - 定义强类型的 Prompt (或使用 OpenAI JSON Schema/Function Calling)，强制 LLM 输出结构化 JSON 数据。
  - 前端以表格或 Checklist 的形式渲染。

## 4. 多文档对比与跨文档查询 (Multi-document QA & Comparison)
**目标：** 解决用户“在多个相关文档中寻找不同点”的痛点。
- **需求描述**：
  - **对比分析**：用户可勾选 2 篇或多篇文档，提问“这两份需求文档中关于登录逻辑的描述有何差异？”。
  - **跨文档聚合回答**：在回答问题时，明确标出信息分别来源于哪几份文档。
- **技术实现路径**：
  - 在检索阶段，支持基于多 `document_id` 的 Filter 进行混合检索。
  - 在 Prompt 中明确要求 LLM 引用文档来源，并对比不同来源的信息。

## 5. UI/UX 体验升级 (UI/UX Enhancements)
- **对话历史持久化**：支持将对话历史保存在本地 (LocalStorage) 或后端数据库中。
- **引用溯源展示 (Citations)**：在 AI 回答下方提供可点击的“来源折叠面板”，展示匹配到的原文段落及页码/来源文件。
