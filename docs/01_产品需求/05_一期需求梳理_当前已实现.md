# 一期需求梳理（当前已实现）

## 1. 一期定位

一期已经完成的是一个可用的“智能文档问答基础版”。

产品主线非常明确：

1. 用户上传文档
2. 系统解析并切片
3. 文档写入向量库
4. 用户基于知识库发起问答
5. 用户在前端查看文档、删除文档、阅读原文预览

当前版本的价值，不是做复杂分析平台，而是先完成“文档入库 + 可问可答 + 可管理”的最小业务闭环。

## 2. 一期目标用户

- 需要快速读懂文档的个人用户
- 需要把 PDF、Word、Markdown 等材料转成可问答知识库的轻量场景
- 希望本地部署检索、云端调用模型、快速跑通流程的团队或开发者

## 3. 一期已实现能力范围

### 3.1 文档上传与格式支持

当前已支持上传并解析以下格式：

- `PDF`
- `DOCX`
- `Markdown`
- `TXT`

能力说明：

- 上传接口已可用
- 对文件类型做了合法性校验
- 对文件大小做了上限控制
- 通过并发信号量限制同时上传数量，避免单机资源被打满

### 3.2 文档解析与入库

当前后端已经完成基础 RAG 入库链路：

- 使用文档解析器读取原文内容
- 为每个文档生成唯一 `doc_id`
- 统一补充 `filename`、`file_type`、`doc_id` 等元数据
- 使用文本切片器按固定 `chunk_size` 和 `chunk_overlap` 分块
- 将分块后的内容写入 ChromaDB

这意味着，一期已经具备“知识库构建”能力，而不是只做临时问答。

### 3.3 文档问答

一期已支持基于知识库进行问答：

- 用户在前端输入问题
- 后端从向量库检索相关分块
- 将检索上下文和用户问题一起发送给 LLM
- 返回中文答案
- 同时返回命中的来源片段与元数据

当前问答特征：

- 默认是面向“当前整个知识库”的问答
- 支持配置检索 `k`
- 已设置请求超时与异常处理
- 回答依赖检索结果，不是纯聊天

### 3.4 文档管理

一期已经实现基础文档管理，不再是纯占位页面：

- 获取文档列表
- 展示文档名称和分块数
- 查看单篇文档详情
- 删除单篇文档

删除逻辑已经打通到底层向量数据：

- 前端发起删除
- 后端按 `doc_id` 删除 ChromaDB 中对应分块
- 删除完成后前端同步更新列表

### 3.5 文档阅读预览

一期已支持文档正文预览：

- 点击文档后可查看文档详情
- 后端按 `doc_id` 聚合文档分块
- 按页码尽量恢复原始阅读顺序
- 前端右侧阅读器展示正文内容

这个能力的价值是让用户在“问答”和“原文”之间来回验证，不必完全黑盒依赖 AI 回答。

### 3.6 基础前端工作台

一期前端已经具备可使用的两大页面：

- `工作台`：当知识库为空时引导上传；有文档后进入问答界面
- `文档管理`：展示文档列表、上传入口、删除能力、文档预览

对话区已实现：

- 多轮消息展示
- 输入与发送
- 加载中状态
- 返回来源文件名标签

### 3.7 系统可用性与部署

一期不是单纯源码堆叠，已经具备基础可部署性：

- 提供 FastAPI 后端服务
- 提供 React 前端服务
- 提供 ChromaDB 服务
- 提供 Docker Compose 一键启动方式
- 提供健康检查接口，检查 `LLM`、`Embedding`、`ChromaDB`
- 支持 `OpenAI Compatible LLM`
- 支持 `OpenAI Compatible Embedding` 与 `Gemini Embedding`

## 4. 一期用户闭环

从产品流程上看，一期已经形成闭环：

1. 进入系统
2. 上传文档
3. 文档解析并入库
4. 在文档管理页查看已入库内容
5. 在工作台基于知识库发起问答
6. 根据来源和正文预览回看原文
7. 删除无效文档，保持知识库干净

## 5. 一期边界与未实现项

为了避免误判范围，需要明确一期没有完成的内容：

- 没有自动摘要生成能力，摘要方法仍是占位
- 没有结构化信息提取，如待办、风险、决策
- 没有按单篇文档或多篇文档做问答范围限定
- 没有多文档对比分析
- 没有会话历史持久化
- 没有账号、团队、权限体系
- 没有批量上传状态管理、批量删除、标签分类等高级管理能力
- 没有精细化引用展示，当前只展示来源文件名，没有高亮定位原文证据

## 6. 一期未完成项与二期承接关系

一期未完成内容，并不等于完全独立的遗留问题。其中一部分会直接进入二期，成为二期核心规划的起点。

可以按以下三类理解：

### 6.1 直接承接到二期的内容

这些能力本质上属于核心链路缺口，二期需要优先补齐：

- `文档摘要`：一期未完成，二期升级为核心需求
- `问答范围控制`：一期只能默认全库问答，二期需要支持单篇、多篇、全库切换
- `引用溯源展示`：一期只展示来源文件名，二期要增强为可查看证据、联动原文
- `会话持久化`：一期没有连续使用能力，二期补齐最近会话保存

### 6.2 以二期增强项形式继续推进的内容

这些内容在一期未完成，但二期不是简单补漏，而是做成更完整的增强能力：

- `多文档对比`：不是单独新增的方向，而是建立在问答范围控制之上的升级能力
- `文档管理补强`：不是改主线，而是让知识库在文档变多后仍然可管理、可维护

### 6.3 暂不进入当前二期主线的内容

这些方向和产品有关，但不建议作为当前二期重点：

- 账号与权限体系
- 团队协作能力
- 复杂批量管理能力

结论上，一期未完成项和二期不是平行关系，而是明显的承接关系。二期的价值，不是重新开新题，而是把一期尚未补齐、但又最影响核心体验的能力做完整。

## 7. 一期验收结论

如果站在产品视角判断，一期已经完成的不是“功能设想”，而是一个可以跑通核心价值的 MVP：

- 能上传
- 能入库
- 能问答
- 能查看
- 能删除

因此，一期需求应被定义为：

> 以“单知识库文档问答”为核心的基础可用版本，重点完成文档入库、知识库问答和基础文档管理闭环。
