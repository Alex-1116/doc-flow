# 二期需求规划（聚焦核心演进）

## 1. 二期目标

二期不建议发散到插件、PR 助手、企业协作等远期方向，而应继续围绕核心主线升级：

> 让用户不仅“能把文档问出来”，还能够“更快读懂文档、明确答案依据、控制问答范围、提升知识库可管理性”。

因此，二期应聚焦在以下四件事：

1. 补齐文档阅读理解能力
2. 补齐问答可信度能力
3. 补齐检索范围控制能力
4. 补齐知识库管理能力

## 2. 二期产品主题

二期建议定义为：

> 智能文档问答增强版

关键词不是“扩品类”，而是“把核心体验做完整”。

## 3. 二期与一期未完成项的关系

二期不是完全独立的新阶段，而是对一期未完成能力的承接和深化。

从规划关系上看，可以拆成两部分：

### 3.1 来自一期未完成项的承接内容

这部分属于二期必须优先处理的核心内容：

- `文档摘要`：一期未完成，二期作为核心能力补齐
- `问答范围控制`：一期默认全库问答，二期补充单篇、多篇、全库切换
- `引用溯源增强`：一期只有来源文件名，二期升级为证据可读、原文可联动
- `会话持久化`：一期没有持续对话能力，二期补齐基本留存体验

### 3.2 在一期基础上继续增强的内容

这部分不是简单收尾，而是在一期核心能力之上继续深入：

- `多文档对比`：建立在范围控制和来源区分基础上的增强能力
- `文档管理补强`：建立在已有上传、列表、删除、预览能力上的可维护性升级

因此，二期更准确的定义应该是：

> 以一期已实现能力为基础，优先承接一期未完成但最影响体验的核心项，再补充少量能够显著增强产品价值的升级项。

## 4. 二期核心需求

### 4.1 文档摘要

#### 目标

让用户在不提问的情况下，也能先快速理解一篇文档的大意和重点。

#### 需求内容

- 支持对单篇文档生成摘要
- 摘要至少包含以下几个部分：
  - 文档主题
  - 核心观点
  - 关键章节 / 结构化大纲
  - 适合快速阅读的简版总结
- 文档上传完成后，用户可主动触发摘要生成
- 在文档详情页或预览区可直接查看摘要结果

#### 产品价值

- 降低“先问什么”的门槛
- 帮助用户快速判断文档是否值得深入阅读
- 让 DocFlow 从“被动回答问题”变成“主动帮助理解文档”

#### 验收口径

- 用户可以对任意已入库文档生成摘要
- 摘要结果可重复查看
- 摘要生成失败时有明确提示

### 4.2 引用溯源增强

#### 目标

让用户知道“答案从哪里来”，提升问答可信度。

#### 需求内容

- 回答结果展示命中的来源片段
- 明确标注来源文档名称
- 若元数据可用，展示页码或段落位置信息
- 用户点击来源后，可联动查看原文预览中的对应内容
- 来源展示不只停留在文件名标签，而是能看到真实证据

#### 产品价值

- 提高用户对回答的信任感
- 支持“先看答案，再核验证据”的阅读方式
- 减少 AI 幻觉带来的误判风险

#### 验收口径

- 每次问答至少展示 Top-K 中的可读来源
- 用户可从答案快速跳转到原文证据

### 4.3 问答范围控制

#### 目标

让用户能够控制本次问答基于哪些文档，而不是默认对全库混检。

#### 需求内容

- 支持选择“单篇文档问答”
- 支持勾选“多篇文档问答”
- 支持“全部知识库问答”作为默认模式
- 回答中明确说明引用来自哪些文档
- 为后续多文档比较打基础

#### 产品价值

- 避免知识库变大后答案被无关文档干扰
- 让问答结果更聚焦、更稳定
- 满足“只针对某份材料提问”的高频场景

#### 验收口径

- 用户可在前端切换问答范围
- 后端检索支持按 `doc_id` 过滤
- 选定范围后，答案来源不应出现范围外文档

### 4.4 多文档对比

#### 目标

解决用户查看多份相似文档时“差异难找”的问题。

#### 需求内容

- 允许用户选择两篇或多篇文档进行对比提问
- 支持典型问题：
  - “这两份需求的差异是什么？”
  - “哪一份文档补充了更多限制条件？”
  - “版本 A 和版本 B 在流程上有什么变化？”
- 输出建议采用结构化形式：
  - 相同点
  - 不同点
  - 仅在某文档中出现的信息

#### 产品价值

- 直接提升知识库使用深度
- 比普通问答更能体现文档工具价值
- 仍然紧扣核心，不引入额外复杂场景

#### 验收口径

- 用户可基于选中文档发起对比问题
- 回答能够按文档来源区分信息

### 4.5 文档管理补强

#### 目标

让知识库从“能用”变成“好管理”。

#### 需求内容

- 文档列表增加更多基础信息：
  - 文件类型
  - 上传时间
  - 处理状态
  - 摘要状态
- 支持搜索文档名称
- 支持按状态或类型筛选
- 删除前给出确认提示
- 为后续批量操作预留能力，但二期不强制做复杂批处理

#### 产品价值

- 提升知识库增长后的可维护性
- 降低误删和找不到文档的问题
- 支撑更稳定的日常使用

#### 验收口径

- 用户可在 50 篇以上文档下快速定位目标文档
- 文档状态清晰可见

### 4.6 会话体验完善

#### 目标

让工作台从“一次性问答框”升级为“可持续使用的阅读对话区”。

#### 需求内容

- 保存最近会话记录
- 新开页面后保留最近一次对话上下文
- 支持清空会话
- 在对话中明确展示当前问答范围

#### 产品价值

- 提高连续提问体验
- 减少用户反复组织上下文的成本

#### 验收口径

- 页面刷新后仍能看到最近会话
- 用户可以主动清空记录

## 5. 二期优先级建议

### P0

- 文档摘要
- 问答范围控制
- 引用溯源增强

### P1

- 多文档对比
- 文档管理补强

### P2

- 会话持久化

说明：

- `P0` 是二期必须完成的核心体验补全
- `P1` 是显著提升产品差异化的能力
- `P2` 可以放在二期后半段实现

## 6. 二期不建议纳入范围

为了保持聚焦，以下内容不建议放入二期：

- 浏览器插件
- PR 助手
- 代码仓库问答
- 会议纪要专项产品化
- 多用户协作与权限体系
- 企业级账号与 SSO

这些方向并非没有价值，而是当前产品仍处于“核心能力打磨期”，过早展开会稀释主线。

## 7. 二期结果定义

如果二期完成，DocFlow 应从“一期可用版本”升级为“真正好用的文档助手”：

- 上传后能快速看摘要
- 问答时能限定范围
- 回答后能查看证据
- 多篇文档能直接比较
- 文档多起来后仍然可管理

因此，二期的本质不是扩张，而是把“文档理解”这条主线做深做透。
