# 📄 DocFlow - 智能文档摘要与问答系统

> **让 AI 帮你读文档。默认轻量部署，支持云端推理与本地检索。**

---

## 🎯 项目概述

### 背景
开发者每天需要阅读大量的技术文档、PR 说明、API 文档、会议记录，但：
- ⏱️ **时间不够**：一篇 50 页文档读完需要 2 小时
- 🧠 **记不住**：读完就忘，需要反复查阅
- 🔍 **找不着**：想找某个细节时需要全文搜索

### 目标
构建一个 **默认轻量、面向云端推理** 的智能文档助手，让 AI 帮你读文档。

---

## ✨ 核心功能

| 模块 | 功能 | 价值 |
|------|------|------|
| 📄 **文档解析** | 支持 PDF、Markdown、Word、代码文件 | 10+ 格式支持 |
| 🗃️ **文档入库** | 文档切片后写入向量库 | 为后续检索和问答提供基础 |
| 💬 **文档问答** | 对着文档提问，AI 精准回答 | 不用全文搜索 |
| 🔍 **语义搜索** | 按意思查找，不是按关键词 | 搜索准确率 ×3 |
| 🩺 **健康检查** | 检查 LLM / Embedding / ChromaDB 状态 | 方便排查服务异常 |

## 💡 技术选型说明

### ✅ 为什么默认走云端推理 + 本地检索？

1. **硬件压力更小** - 低配机器不再常驻本地大模型，避免 CPU 打满
2. **保留 RAG 架构** - 文档解析与向量检索仍在本地，改造成本低
3. **接入灵活** - 支持 OpenAI 兼容 LLM 与 Gemini/OpenAI 兼容 Embedding
4. **部署更稳** - 默认只需要启动前后端和 ChromaDB，冷启动更快
5. **适合演进** - 先跑通业务，再按服务商能力扩展不同 provider

### ❌ 为什么不再依赖本地大模型？

- 硬件门槛高：低配机器运行本地大模型容易导致 CPU 和内存爆满
- 首次体验差：本地模型准备时间长，冷启动慢
- 稳定性受限：本地资源不足时，检索和问答链路容易整体超时

### ✅ 当前已实现

1. **多格式文档解析** - 支持常见文档格式上传和解析
2. **向量化入库** - 文档切片后写入 ChromaDB
3. **RAG 文档问答** - 基于检索结果生成回答
4. **服务健康检查** - 可查看 LLM / Embedding / ChromaDB 状态
5. **基础前端界面** - 支持上传文件与发起问答

### 🧪 规划中

1. **文档摘要** - 当前仍为占位能力，尚未完成
2. **要点提取** - Action Items / 风险 / 决策提取尚未落地
3. **多文档对比** - 尚未实现跨文档差异分析
4. **完整文档列表** - 当前列表接口尚未返回真实数据
5. **扩展工具能力** - 浏览器插件、会议纪要、PR 助手等仍在规划

## 🚀 快速开始

```bash
# 克隆项目
git clone https://github.com/Alex-1116/doc-flow.git
cd doc-flow

# 复制环境变量模板
cp .env.example .env

# 编辑 .env，填入 LLM / Embedding 配置

# 构建服务镜像
docker compose build
# 启动默认服务（frontend + backend + chromadb）
docker compose up -d
# 停止服务
docker compose down
# 查看服务状态
docker compose ps
# 重启服务
docker compose restart

# 热更配置 
docker-compose.override.yml

# 打开浏览器
open http://localhost:3000
```
